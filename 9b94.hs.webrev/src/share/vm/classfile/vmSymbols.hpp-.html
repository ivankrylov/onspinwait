 622 // Here are all the intrinsics known to the runtime and the CI.
 623 // Each intrinsic consists of a public enum name (like _hashCode),
 624 // followed by a specification of its klass, name, and signature:
 625 //    template(<id>,  <klass>,  <name>, <sig>, <FCODE>)
 626 //
 627 // If you add an intrinsic here, you must also define its name
 628 // and signature as members of the VM symbols.  The VM symbols for
 629 // the intrinsic name and signature may be defined above.
 630 //
 631 // Because the VM_SYMBOLS_DO macro makes reference to VM_INTRINSICS_DO,
 632 // you can also define an intrinsic's name and/or signature locally to the
 633 // intrinsic, if this makes sense.  (It often does make sense.)
 634 //
 635 // For example:
 636 //    do_intrinsic(_foo,  java_lang_Object,  foo_name, foo_signature, F_xx)
 637 //     do_name(     foo_name, "foo")
 638 //     do_signature(foo_signature, "()F")
 639 // klass      = vmSymbols::java_lang_Object()
 640 // name       = vmSymbols::foo_name()
 641 // signature  = vmSymbols::foo_signature()
 642 //
 643 // The name and/or signature might be a "well known" symbol
 644 // like "equal" or "()I", in which case there will be no local
 645 // re-definition of the symbol.
 646 //
 647 // The do_class, do_name, and do_signature calls are all used for the
 648 // same purpose:  Define yet another VM symbol.  They could all be merged
 649 // into a common 'do_symbol' call, but it seems useful to record our
 650 // intentions here about kinds of symbols (class vs. name vs. signature).
 651 //
 652 // The F_xx is one of the Flags enum; see below.
 653 //
 654 // for Emacs: (let ((c-backslash-column 120) (c-backslash-max-column 120)) (c-backslash-region (point) (point-max) nil t))
 655 //
 656 //
 657 // There are two types of intrinsic methods: (1) Library intrinsics and (2) bytecode intrinsics.
 658 //
 659 // (1) A library intrinsic method may be replaced with hand-crafted assembly code,
 660 // with hand-crafted compiler IR, or with a combination of the two. The semantics
 661 // of the replacement code may differ from the semantics of the replaced code.
 662 //
 663 // (2) Bytecode intrinsic methods are not replaced by special code, but they are
 664 // treated in some other special way by the compiler. For example, the compiler
 665 // may delay inlining for some String-related intrinsic methods (e.g., some methods
 666 // defined in the StringBuilder and StringBuffer classes, see
 667 // Compile::should_delay_string_inlining() for more details).
 668 //
 669 // Due to the difference between the semantics of an intrinsic method as defined
 670 // in the (Java) source code and the semantics of the method as defined
 671 // by the code in the VM, intrinsic methods must be explicitly marked.
 672 //
 673 // Intrinsic methods are marked by the jdk.internal.HotSpotIntrinsicCandidate
 674 // annotation. If CheckIntrinsics is enabled, the VM performs the following
 675 // checks when a class C is loaded: (1) all intrinsics defined by the VM for
 676 // class C are present in the loaded class file and are marked;
 677 // (2) an intrinsic is defined by the VM for all marked methods of class C;
 678 // (3) check for orphan methods in class C (i.e., methods for which the VM
 679 // declares an intrinsic but that are not declared for the loaded class C.
 680 // Check (3) is available only in debug builds.
 681 //
 682 // If a mismatch is detected for a method, the VM behaves differently depending
 683 // on the type of build. A fastdebug build exits and reports an error on a mismatch.
 684 // A product build will not replace an unmarked library intrinsic method with
 685 // hand-crafted code, that is, unmarked library intrinsics are treated as ordinary
 686 // methods in a product build. The special treatment of a bytecode intrinsic method
 687 // persists even if the method not marked.
 688 //
 689 // When adding an intrinsic for a method, please make sure to appropriately
 690 // annotate the method in the source code. The list below contains all
 691 // library intrinsics followed by bytecode intrinsics. Please also make sure to
 692 // add the declaration of the intrinsic to the approriate section of the list.
 693 #define VM_INTRINSICS_DO(do_intrinsic, do_class, do_name, do_signature, do_alias)                                       \
 694   /* (1) Library intrinsics                                                                        */                   \
 695   do_intrinsic(_hashCode,                 java_lang_Object,       hashCode_name, void_int_signature,             F_R)   \
 696    do_name(     hashCode_name,                                   "hashCode")                                            \
 697   do_intrinsic(_getClass,                 java_lang_Object,       getClass_name, void_class_signature,           F_R)   \
