 657 // Here are all the intrinsics known to the runtime and the CI.
 658 // Each intrinsic consists of a public enum name (like _hashCode),
 659 // followed by a specification of its klass, name, and signature:
 660 //    template(<id>,  <klass>,  <name>, <sig>, <FCODE>)
 661 //
 662 // If you add an intrinsic here, you must also define its name
 663 // and signature as members of the VM symbols.  The VM symbols for
 664 // the intrinsic name and signature may be defined above.
 665 //
 666 // Because the VM_SYMBOLS_DO macro makes reference to VM_INTRINSICS_DO,
 667 // you can also define an intrinsic's name and/or signature locally to the
 668 // intrinsic, if this makes sense.  (It often does make sense.)
 669 //
 670 // For example:
 671 //    do_intrinsic(_foo,  java_lang_Object,  foo_name, foo_signature, F_xx)
 672 //     do_name(     foo_name, "foo")
 673 //     do_signature(foo_signature, "()F")
 674 // klass      = vmSymbols::java_lang_Object()
 675 // name       = vmSymbols::foo_name()
 676 // signature  = vmSymbols::foo_signature()
 677 //
 678 // The name and/or signature might be a "well known" symbol
 679 // like "equal" or "()I", in which case there will be no local
 680 // re-definition of the symbol.
 681 //
 682 // The do_class, do_name, and do_signature calls are all used for the
 683 // same purpose:  Define yet another VM symbol.  They could all be merged
 684 // into a common 'do_symbol' call, but it seems useful to record our
 685 // intentions here about kinds of symbols (class vs. name vs. signature).
 686 //
 687 // The F_xx is one of the Flags enum; see below.
 688 //
 689 // for Emacs: (let ((c-backslash-column 120) (c-backslash-max-column 120)) (c-backslash-region (point) (point-max) nil t))
 690 //
 691 //
 692 // There are two types of intrinsic methods: (1) Library intrinsics and (2) bytecode intrinsics.
 693 //
 694 // (1) A library intrinsic method may be replaced with hand-crafted assembly code,
 695 // with hand-crafted compiler IR, or with a combination of the two. The semantics
 696 // of the replacement code may differ from the semantics of the replaced code.
 697 //
 698 // (2) Bytecode intrinsic methods are not replaced by special code, but they are
 699 // treated in some other special way by the compiler. For example, the compiler
 700 // may delay inlining for some String-related intrinsic methods (e.g., some methods
 701 // defined in the StringBuilder and StringBuffer classes, see
 702 // Compile::should_delay_string_inlining() for more details).
 703 //
 704 // Due to the difference between the semantics of an intrinsic method as defined
 705 // in the (Java) source code and the semantics of the method as defined
 706 // by the code in the VM, intrinsic methods must be explicitly marked.
 707 //
 708 // Intrinsic methods are marked by the jdk.internal.HotSpotIntrinsicCandidate
 709 // annotation. If CheckIntrinsics is enabled, the VM performs the following
 710 // checks when a class C is loaded: (1) all intrinsics defined by the VM for
 711 // class C are present in the loaded class file and are marked;
 712 // (2) an intrinsic is defined by the VM for all marked methods of class C;
 713 // (3) check for orphan methods in class C (i.e., methods for which the VM
 714 // declares an intrinsic but that are not declared for the loaded class C.
 715 // Check (3) is available only in debug builds.
 716 //
 717 // If a mismatch is detected for a method, the VM behaves differently depending
 718 // on the type of build. A fastdebug build exits and reports an error on a mismatch.
 719 // A product build will not replace an unmarked library intrinsic method with
 720 // hand-crafted code, that is, unmarked library intrinsics are treated as ordinary
 721 // methods in a product build. The special treatment of a bytecode intrinsic method
 722 // persists even if the method not marked.
